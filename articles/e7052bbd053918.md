## 概要

こんにちは。今回は Azure SRE Agent を触ってみます。
Azure SRE Agent は、AI エージェントサービスで、トラブルシューティング支援や自動化を支援してくれるものです。
実は Azure 内部のトラブルシューティングにも利用されています。

今回、SRE Agent を使ってトラブルシューティングの内容を文書化してもらうというシナリオを試してみます。
トラブルシューティングの内容は、実施時にまとめたり、後から思い出して作成することがなかなか大変だと思います。
大変な割には、そんなに見直されないドキュメントは容易に陳腐化してしまいます。

SRE Agent を使うとそのトラブルシューティングを支援するだけでなく、最後に振り返りを行って、
ナレッジ化し、次回その情報を参照させることができます。

## 現時点の制限

2026年3月10日に、Azure SRE Agent は GA となりました [^1] が引き続き以下のような制限があります。
- 利用可能リージョン: East US 2, Sweden Central, Australia East
- 言語: 英語のみ
- 対応サービス: 一部の Azure サービスに限定 (VM、App Service、Azure SQL など)

## SRE Agent の設定

**初期セットアップ:**
SRE Agent ポータルを開くと下記のような初期セットアップ ウィザードが表示されます。
後から再度この画面の表示も可能なため、一旦 "Done and go to agent" とします。

![alt text](/images/e7052bbd053918/sreagt1.png)

**Azure Monitor との統合:**

インデント プラットフォームが既定では選択されていない状態のため、Azure Monitor を設定します。
左メニューの "Builder" → "インシデント プラットフォーム" を選択し、"Azure Monitor" を選択して保存します。
その際、下記の用に Azure Monitor の Sev3 アラートに対応するインシデント対応計画を自動で作成するオプションがありますが、
今回はオフにします。

![alt text](/images/e7052bbd053918/sreagt2.png)

**サブエージェントの作成:**

次にサブエージェントを作ります。これによりエージェントのふるまいをカスタマイズすることが可能です。
以前は既定ではナレッジベースの参照や更新ができませんでしたが、GA 後は既定でナレッジベースを管理するツールが継承され、有効になっています。

![alt text](/images/e7052bbd053918/sreagt3.png)

1. Azure SRE Agent ポータルで `srelab-agent` を選択し、"Builder" → "Agent Canvas" タブを選択します。
2. "+ 作成" をクリックします。
3. "サブエージェントを作成する":
   - サブエージェント名: `srelab-subagent`
   - 指示: `You are an Azure Monitor Alert specialist responsible for managing and responding to 
Azure Monitor alerts. When an alert fires, you investigate the affected resource by 
analyzing metrics, logs, and recent changes. You diagnose the root cause, propose 
mitigation steps, and document findings. Use az cli and Log Analytics queries to 
gather diagnostics. If the issue requires resource modification, propose changes for 
review before executing.`

**インシデント対応計画:**

1. Azure SRE Agent ポータルで `srelab-agent` を選択し、"Builder" → "Agent Canvas" を選択します。
2. "+ 作成" → "トリガー" → "インシデント応答プラン" を選択します。
3. "対応計画:
    - インシデント対応計画名: `srelab-incident-plan`
    - 重要度: すべての重要度
    - 応答サブエージェント: `srelab-subagent`
    - エージェントの自律性レベル: `Autonomous(Default)`
4. "作成" をクリックして保存します。

![alt text](/images/e7052bbd053918/sreagt4.png)

## SRE Agent によるトラブルシューティング

実験を開始してから数分後、Azure SRE Agent ポータル にインシデントが作成されます。

1. Azure SRE Agent ポータルで `srelab-agent` を選択し、"アクティビティ" → "インシデント" タブを選択します。

![alt text](/images/e7052bbd053918/sreagt5.png)

2. SRE Agent の方でアラートを契機に調査が自動実行されています。アラートを受けてインシデント応答の計画が作られています。

まずは、SRE Agent がアラートを受け取り、インシデントの包括的な対応方針について計画します。

![alt text](/images/e7052bbd053918/sreagt6.png)

その後、最終的な対応方針が決まります。
今回は、以下のようにメトリックの分析を行った後、そのようなプロセスがあればそのプロセスを停止する方針となりました。また、Chaos Studio による実験の可能性も指摘されています (バレてますね...)。

![alt text](/images/e7052bbd053918/sreagt7.png)

```text
1. Gather VM details — Get VM properties, power state, and OS info（VM の詳細を収集 — VM のプロパティ、電源状態、OS 情報を取得）
2. Analyze CPU metrics — Pull recent CPU percentage time-series to confirm severity and trend（CPU メトリクスを分析 — 直近の CPU 使用率の時系列データを取得し、緊急度と傾向を確認）
3. Check activity logs — Look for recent changes, deployments, or Chaos Studio experiments（アクティビティ ログを確認 — 最近の変更、デプロイ、または Chaos Studio の実験を調査）
4. Identify CPU consumers — If VM is accessible, run diagnostics to find top processes（CPU 消費元を特定 — VM にアクセス可能であれば、診断を実行して上位プロセスを特定）
5. Mitigate — Kill runaway processes or stop chaos experiment if applicable（軽減 — CPU 使用率が高いプロセスの停止、または該当する場合は Chaos 実験を停止）
6. Summarize — Document root cause and actions taken（要約 — 根本原因と実施したアクションを文書化）
```

案の定、メトリック傾向を確認したのち、Azure アクティビティログを確認した結果、Chaos Studio より CPU 負荷の実験が見つかったため、それによる影響という判断に至りました。

![alt text](/images/e7052bbd053918/sreagt8.png)

その後、再発がないことを確認し、最終的な調査報告が行われます。

![alt text](/images/e7052bbd053918/sreagt9.png)

## SRE Agent にトラブルシューティングの内容をドキュメント化を依頼

実施したインシデントについてドキュメントに残すように指示します。

これによって、今回のインシデント対応の記録がナレッジ ベースに保存されます。
また、フォーマットが決まって入れば、書く人によって内容が異なるといったことがなく、一貫したドキュメントが作成可能です。

```text
We've confirmed that the high CPU issue on the VM has been resolved. Based on this incident, create a remediation runbook in Markdown file and upload it to knowledge sources.
```

![alt text](/images/e7052bbd053918/sreagt10.png)

プレビューでは、エージェントにドキュメントの内容を貼り付けてもらう必要があったのですが、GA 後ではドキュメントのリンクｇ生成されるようになっています。ただし、現時点で試した段階では、ドキュメントが作成され、以降の対応時に参照されるものの、"Builder" → "Knowlege sources" には保存されず、ユーザーメモリに保存されています。

![alt text](/images/e7052bbd053918/sreagt11.png)

また、同様のインシデントが発生した際に、その対応に沿って自動的に更新してくれます。

![alt text](/images/e7052bbd053918/sreagt12.png)

::::details ドキュメントの内容

```markdown
# High CPU Alert Remediation Runbook — Azure Virtual Machines

> **Last updated:** 2026-03-26  
> **Derived from:** Incidents on `srelab-vm` (2026-03-19, 2026-03-26) — Chaos Studio CPU pressure experiments triggered Sev2 alerts.

---

## 1. Alert Context

| Field | Expected Value |
|---|---|
| Alert rule pattern | `*-high-cpu-alert` |
| Metric | `Percentage CPU` |
| Threshold | > 80% sustained for ≥ 5 minutes |
| Severity | Sev2 |
| Resource type | `microsoft.compute/virtualmachines` |

---

## 2. Triage Checklist

Run these steps **in order** upon receiving the alert. Stop early if root cause is confirmed.

### Step 1 — Confirm VM State

```bash
az vm show --name <VM_NAME> --resource-group <RG> --show-details \
  --query "{name:name, powerState:powerState, vmSize:hardwareProfile.vmSize, osType:storageProfile.osDisk.osType}" \
  -o json
```

- If `powerState` is not `VM running`, the alert may be stale — verify and close.
- Note the `vmSize` (e.g., `Standard_B2s`) — burstable SKUs have CPU credit limits that can amplify perceived high CPU.

### Step 2 — Validate CPU Metrics

Pull the last 30 minutes at 1-minute granularity to understand the spike shape:

```bash
az monitor metrics list \
  --resource <VM_RESOURCE_ID> \
  --metric "Percentage CPU" \
  --interval PT1M \
  --start-time $(date -u -d '30 minutes ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --aggregation Average Maximum \
  --query "value[0].timeseries[0].data[]" -o table
```

**Key patterns to look for:**

| Pattern | Likely Cause |
|---|---|
| Sudden step-function jump to a fixed level (e.g., ~82%) | Stress test or Chaos Studio experiment |
| Gradual ramp over hours | Organic load increase or memory leak causing swap thrashing |
| Periodic spikes aligned with cron schedule | Scheduled job (backup, log rotation, batch processing) |
| Sustained 100% with no drop | Runaway process, infinite loop, or resource exhaustion |

### Step 3 — Check for Chaos Studio Experiments

**This is a critical early check.** The alert description may reference "Chaos Studio demo" or similar. Always verify:

```bash
az monitor activity-log list \
  --resource-group <RG> \
  --start-time $(date -u -d '2 hours ago' +%Y-%m-%dT%H:%M:%SZ) \
  --query "[?contains(operationName.value, 'Microsoft.Chaos')].{time:eventTimestamp, operation:operationName.value, status:status.value, caller:caller, resourceId:resourceId}" \
  -o table
```

If Chaos experiments are found, get execution status:

```bash
az rest --method get \
  --url "https://management.azure.com/subscriptions/<SUB_ID>/resourceGroups/<RG>/providers/Microsoft.Chaos/experiments/<EXPERIMENT_NAME>/executions?api-version=2024-01-01" \
  --query "value[0].{status:properties.status, startedAt:properties.startedAt, stoppedAt:properties.stoppedAt}" \
  -o json
```

**If experiment is still running and needs to be stopped immediately:**

```bash
az rest --method post \
  --url "https://management.azure.com/subscriptions/<SUB_ID>/resourceGroups/<RG>/providers/Microsoft.Chaos/experiments/<EXPERIMENT_NAME>/cancel?api-version=2024-01-01"
```

**If experiment has completed (status: `Success`):**
- CPU should recover within 1–2 minutes of experiment completion.
- Monitor metrics for 3–5 minutes to confirm recovery.
- No further action needed — proceed to documentation.

### Step 4 — Check Activity Logs for Other Changes

If no Chaos experiments found, look for recent configuration changes:

```bash
az monitor activity-log list \
  --resource-group <RG> \
  --start-time $(date -u -d '4 hours ago' +%Y-%m-%dT%H:%M:%SZ) \
  --query "[?contains(operationName.value, 'write') || contains(operationName.value, 'Write')].{time:eventTimestamp, operation:operationName.value, status:status.value, caller:caller}" \
  -o table
```

Look for: VM restarts, extension installations, image changes, scale operations, or NSG modifications.

### Step 5 — Identify Top CPU Consumers on the VM

If the VM is accessible via SSH or Run Command:

```bash
az vm run-command invoke \
  --resource-group <RG> --name <VM_NAME> \
  --command-id RunShellScript \
  --scripts "ps aux --sort=-%cpu | head -15"
```

For Windows VMs:

```bash
az vm run-command invoke \
  --resource-group <RG> --name <VM_NAME> \
  --command-id RunPowerShellScript \
  --scripts "Get-Process | Sort-Object CPU -Descending | Select-Object -First 15 Name, Id, CPU, WorkingSet"
```

**Common culprits:**

| Process | Action |
|---|---|
| `stress` / `stress-ng` | Chaos Studio or manual stress test — will self-resolve or kill |
| `python` / `java` / `node` with high CPU | Application issue — check logs, consider restart |
| `apt` / `yum` / `dpkg` | Package update in progress — wait for completion |
| `waagent` / `WaLinuxAgent` | Azure agent issue — check `/var/log/waagent.log` |
| `mdsd` / `OMSAgent` | Monitoring agent issue — restart the agent |

---

## 3. Mitigation Actions

### 3a — Chaos Studio Experiment (Self-Resolving)

No action required if:
- Experiment status is `Success` (already completed)
- CPU returns to baseline within 2 minutes

Cancel if experiment is still running and not expected:
```bash
az rest --method post \
  --url "https://management.azure.com/subscriptions/<SUB_ID>/resourceGroups/<RG>/providers/Microsoft.Chaos/experiments/<EXPERIMENT_NAME>/cancel?api-version=2024-01-01"
```

### 3b — Runaway Process

Kill the offending process:
```bash
az vm run-command invoke \
  --resource-group <RG> --name <VM_NAME> \
  --command-id RunShellScript \
  --scripts "kill -9 <PID>"
```

### 3c — Organic Load Increase

1. Check if autoscale is configured — if not, consider enabling it.
2. For immediate relief, scale up the VM:
   ```bash
   az vm resize --resource-group <RG> --name <VM_NAME> --size <LARGER_SKU>
   ```
   > **Note:** This will restart the VM. Ensure this is acceptable.
3. For burstable VMs (`Standard_B*`), check if CPU credits are exhausted:
   ```bash
   az monitor metrics list --resource <VM_RESOURCE_ID> \
     --metric "CPU Credits Remaining" --interval PT5M \
     --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ) \
     --aggregation Average --query "value[0].timeseries[0].data[-5:]" -o table
   ```

### 3d — VM Restart (Last Resort)

If the VM is unresponsive and no other mitigation works:
```bash
az vm restart --resource-group <RG> --name <VM_NAME>
```

---

## 4. Verification

After mitigation, confirm CPU has recovered:

```bash
az monitor metrics list \
  --resource <VM_RESOURCE_ID> \
  --metric "Percentage CPU" \
  --interval PT1M \
  --start-time $(date -u -d '10 minutes ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --aggregation Average Maximum \
  --query "value[0].timeseries[0].data[]" -o table
```

**Success criteria:** Average CPU below 80% for at least 5 consecutive minutes.

> **Note:** Azure Monitor metrics have a 1–2 minute ingestion delay. In practice, wait **~3 minutes** after expected recovery before querying verification metrics to ensure data points are populated.

---

## 5. Post-Incident Recommendations

### Alert Tuning
- Add action group suppression during scheduled Chaos Studio test windows.
- Consider a secondary alert at 95% (Sev1) for genuine production emergencies.
- For burstable VMs, also alert on `CPU Credits Remaining < 10`.

### Chaos Studio Best Practices
- Notify on-call teams before running experiments via a shared channel or calendar entry.
- Tag experiments and test windows so alert rules can filter them.
- Keep experiment durations short (≤ 10 min) for CPU pressure tests.
- Always verify the experiment has a finite duration (`PT*M` or `PT*H`) — never run open-ended.

### Operational Hygiene
- Document all Chaos experiments in a shared runbook or wiki.
- Ensure the `Microsoft-Agent` Chaos target has appropriate RBAC scoped only to test VMs.
