こんにちは。今回は Azure SRE Agent を触ってみます。
Azure SRE Agent は、AI エージェントサービスで、トラブルシューティング支援や自動化を支援してくれるものです。
実は Azure 内部のトラブルシューティングにも利用されています。

今回、SRE Agent を使ってトラブルシューティングの内容を文書化してもらうというシナリオを試してみます。
トラブルシューティングの内容は、実施時にまとめたり、後から思い出して作成することがなかなか大変だと思います。
大変な割には、そんなに見直されないドキュメントは容易に陳腐化してしまいます。

SRE Agent を使うとそのトラブルシューティングを支援するだけでなく、最後に振り返りを行って、
ナレッジ化し、次回その情報を参照させることができます。

## 現時点の制限

2026年3月時点では、Azure SRE Agent はプレビュー段階で、利用できるリージョンや機能に制限があります。
- 利用可能リージョン: East US 2, West Europe, Southeast Asia
- 言語: 英語のみ
- 対応サービス: 一部の Azure サービスに限定 (VM、App Service、Azure SQL など)

### SRE Agent の作成

> 現在 Azure SRE Agent ポータル (https://sre.azure.com/) にて操作を行うようになっています。

1. Azure SRE Agent ポータル にアクセスし、Azure アカウントでサインインします。
2. "+ エージェントの作成" をクリックします。
3. "基本情報":
   - サブスクリプション: 使用するサブスクリプションを選択
   - リソース グループ: `rg-sreagent` (SRE Agent 用のリソース グループを新規作成)
   - エージェント名: `srelab-agent`
   - リージョン: `East US 2`
   - Application Insights: `新規作成` を選択
4. "管理対象 リソース グループ":
   - `rg-srelab` を選択し、"次へ" をクリックします。
5. "エージェントのアクセス許可":
   - "特権" を選択し、"次へ" をクリックします。
7. "作成" をクリックしてエージェントを作成します。

## SRE Agent の設定

**Azure Monitor との統合:**

既定では Azure Monitor がインシデント プラットフォームとして設定されているはずですが、
画面ではされていないように見えます (設定するとエラーになるので、一旦はこのまま進めます)。

![alt text](sreagt1.png)

**サブエージェントの作成:**

次にサブエージェントを作ります。これによりエージェントのふるまいをカスタマイズすることが可能です。
その一つに、既定のエージェントでは、ナレッジベースへのアクセスが行えないため、今回のシナリオではサブエージェントを作成し、ナレッジベースへのアクセス権を付与します。

1. Azure SRE Agent ポータルで `srelab-agent` を選択し、"Builder" → "サブエージェント ビルダー" タブを選択します。
2. "+ 作成" をクリックします。
3. "サブエージェントを作成する":
   - サブエージェント名: `srelab-subagent`
   - 指示: `You are an Azure Monitor Alert specialist responsible for managing and responding to 
Azure Monitor alerts. When an alert fires, you investigate the affected resource by 
analyzing metrics, logs, and recent changes. You diagnose the root cause, propose 
mitigation steps, and document findings. Use az cli and Log Analytics queries to 
gather diagnostics. If the issue requires resource modification, propose changes for 
review before executing.`
4. "Tool" では今回下記を利用
   - GetArmResourceAsJson
   - GetAzCliHelp
   - PowerOnVirtualMachine
   - RunAzCliReadCommands
   |- RunAzCliWriteCommands
   - "Diagnostics" カテゴリ
   - AnalyzeDeploymentFailures
   - GetActivityLogsSummary
   - GetChangeHistory
   - SearchIncidentKnowledge
   - SearchMemory
   - UploadKnowledgeDocument (← ナレッジベースに保存するため付与)
   - GetMetricTimeSeriesElementsForAzureResource
   - ListAvailableMetrics
   - QueryLogAnalyticsByResourceId
   - "Visualization" カテゴリ

5. "ナレッジ ベースへのアクセス権を付与する" を有効化

**インシデント対応計画:**

1. Azure SRE Agent ポータルで `srelab-agent` を選択し、"Builder" → "サブエージェント ビルダー" を選択します。
2. "+ 作成" → "インシデント応答プラン" を選択します。
3. "対応計画:
    - インシデント対応計画名: `srelab-incident-plan`
    - 重要度: すべての重要度
    - エージェントの自律性レベル: `Autonomous(Default)`
4. "作成" をクリックして保存します。

## Chaos Studio を使って、障害をシミュレート

**実験の作成:**

1. Azure ポータルで Chaos Studio を開き、"実験" タブを選択します。
2. "+ 作成" → "新しい実験" をクリックします。
3. "基本情報" タブ:

| 設定項目 | 値 |
|---|---|
| リソース グループ | `rg-srelab` |
| 名前 | `srelab-cpu-pressure-exp` |
| リージョン | `East US 2` |

3. "アクセス許可" タブ:
   - マネージド ID: `umi-srelab` を選択
   - 実験のアクセス許可: `Azure 組み込みロールを使用する` を選択

4. "実験デザイナー" タブ:
   - ステップにて、`step1-cpu-pressure` を作成します。
   - ブランチを `branch1` にします。
   - "+ アクションの追加" をクリックします。

　 | 設定項目 | 値 |
   |---|---|
   | 障害 | `CPU Pressure` |
   | 期間 | `20 分` |
   | pressureLevel | `95` |
   
   - "次へ: ターゲット リソース" で `vm-srelab` を選択し、"追加" をクリックします。
   - "フェイル シナリオの選択" で "CPU Pressure" を選択します。
   - "シナリオの構成" で以下を設定します。

5. "確認と作成" タブで内容を確認し、"作成" をクリックします。

**実験の実行:**

1. Chaos Studio の "実験" にて作成した実験 `srelab-cpu-pressure-exp` を選択します。
2. "実験の開始" をクリックし、次いで "はい" をクリックして実験を開始します。

徐々に CPU 使用率が上がって行き、しばらく待つとアラートが発報されます。

## SRE Agent によるトラブルシューティング

実験を開始してから数分後、Azure SRE Agent ポータル にインシデントが作成されます。

1. Azure SRE Agent ポータルで `srelab-agent` を選択し、"アクティビティ" → "インシデント" タブを選択します。

![alt text](sreagt5.png)

2. SRE Agent の方でアラートを契機に調査が自動実行されています。アラートを受けてインシデント応答の計画が作られています。

まずは、SRE Agent がアラートを受け取り、インシデントの包括的な対応方針について計画します。

![alt text](sreagt6.png)

対応方針に従い、VM のメトリックが確認され、緩和策として VM の再起動が実施されることが決まりました。その後、VM 再起動が実施され、緩和策の効果を評価します。

![alt text](sreagt7.png)

その後、再発がないことを確認し、対応を終了します。

```text
High-CPU alert mitigated. Findings: CPU sustained ~83% from 08:39–08:47 UTC. Action: Restarted srelab-vm; it’s running and CPU dropped to ~0.7% from 08:49 UTC onward. I’ll watch CPU every 5 minutes for the next 60 minutes and notify if it exceeds 80% again. One follow-up: if you share your Log Analytics workspace ID, I can enable VM Insights to capture top processes for root-cause analysis in the future.
```

## SRE Agent にトラブルシューティングの内容をドキュメント化を依頼

実施したインシデントについてドキュメントに残すように指示します。

これによって、今回のインシデント対応の記録がナレッジ ベースに保存されます。
また、フォーマットが決まって入れば、書く人によって内容が異なるといったことがなく、一貫したドキュメントが作成可能です。

![alt text](sreagt8.png)
![alt text](sreagt9.png)

また、保存したナレッジの更新を依頼することも可能です。

![alt text](sreagt10.png)

ただ一点、現時点ではナレッジ ベースからダウンロードする方法がなく、エージェントに取得をお願いする必要がある状況でした。

作成されたドキュメントはこちらです。

::::details ドキュメントの内容

```markdown
Azure VM High CPU Alert – Remediation Runbook

Version: 2026-03-04
Owner: SRE
Related alert: srelab-high-cpu-alert (CPU > 80% for 5m) 
Scope: Azure IaaS VMs (Linux/Windows)

Purpose 

Provide a fast, safe path to triage and mitigate sustained high CPU on a VM, with clear verification, rollback, and known pitfalls.

Preconditions

- Azure CLI logged in with appropriate RBAC
- Resource identifiers handy: subscription, resource group, VM name or resource ID

Quick Triage (5–10 minutes)

1. Confirm alert window and baseline
- Metrics (1m granularity, last 30–60m): Percentage CPU
- Correlate with Disk Read/Write Operations/Sec and Network In/Out if needed
2. Identify VM characteristics
- Size/family, region, OS, extensions, power state
3. For B-series (burstable) VMs, check CPU credits
- If credits depleted, expect throttling; mitigation is scale or wait for credits
4. Look for recent changes
- Activity Log: deployments, RunCommand, resize/stop/start, extension updates
5. Check guest workload
- Identify top processes (SSH/RDP/RunCommand); check schedulers/cron/Task Scheduler, background scans, or deployments

Azure CLI – Triage Commands Replace variables accordingly: SUB, RG, VM

- VM properties az vm show -g $RG -n $VM --subscription $SUB -o jsonc
- Power state / instance view az vm get-instance-view -g $RG -n $VM --subscription $SUB --query "instanceView.statuses[].code" -o tsv

- Percentage CPU (last 30 minutes, 1m) az monitor metrics list --subscription $SUB
--resource "/subscriptions/$SUB/resourceGroups/$RG/providers/Microsoft.Compute/virtualMachines/$VM"
--metric "Percentage CPU" --interval PT1M --aggregation Average
--start-time "30 minutes ago" --end-time "now"

- CPU Credits Remaining (B-series only) az monitor metrics list --subscription $SUB
--resource "/subscriptions/$SUB/resourceGroups/$RG/providers/Microsoft.Compute/virtualMachines/$VM"
--metric "CPU Credits Remaining" --interval PT1M --aggregation Average
--start-time "2 hours ago" --end-time "now"

- Activity Log (last 3 hours) az monitor activity-log list --subscription $SUB --resource-group $RG
--offset 3h --status Succeeded -o table

- Available resize options az vm list-vm-resize-options -g $RG -n $VM --subscription $SUB -o table

Guest OS Checks

- Linux:
   - top -b -n1 | head -40
   - pidstat -p ALL 1 5
   - mpstat -P ALL 1 5; vmstat 1 10
   - journalctl -S "30 min ago"
- Windows:
   - Get-Process | Sort CPU -Desc | Select -First 20
   - Get-WmiObject Win32_PerfFormattedData_PerfOS_Processor
   - Event Viewer: System/Application around alert time

Mitigation Options Choose the least disruptive first. Ensure service windows for production.

A) Restart VM (short outage; frees stuck processes) az vm restart -g $RG -n $VM --subscription $SUB

B) Stop/Start (reallocate host; may change dynamic IP)

Stop (allocating stays billed) or deallocate (release compute) az vm stop -g $RG -n $VM --subscription $SUB
or
az vm deallocate -g $RG -n $VM --subscription $SUB

Start az vm start -g $RG -n $VM --subscription $SUB
C) Resize VM (increase vCPU/memory or move off burstable)

1. Enumerate options az vm list-vm-resize-options -g $RG -n $VM --subscription $SUB -o table
2. If VM must be deallocated before resize (common): az vm deallocate -g $RG -n $VM --subscription $SUB
3. Resize to available SKU (handle SkuNotAvailable by picking an alternative from step 1) az vm resize -g $RG -n $VM --size Standard_D2s_v3 --subscription $SUB
```
::::

## 既定のエージェントの場合

既定のエージェントに対して同様にトラブルシューティング後に、ドキュメントをナレッジ ベースに追加してもらうようお願いしてみます。

```text
Please document the response to this incident as a runbook.
```

手順書 (Runbook) を作成してもらえまずが、ナレッジベースへの保存はできないようです。そのため、サブエージェントを作成するのがまずは必要でした。

```text
Please store the generated runbook in the knowledge base as a Markdown file.
```

## おわりに

今回は Azure SRE Agent を使って、Azure Monitor のアラートをトリガーとしたインシデントに対し、SRE Agent にトラブルシューティングを行ってもらい、その内容をナレッジベースに保存するところを試してみました。このナレッジ ベースの利用フローについては公式ドキュメントに詳しいので、そちらも参照ください [^1]。

なお、組織で横断的にナレッジを活用するのであれば、Azure DevOps Wiki や GitHub に集約し、SRE Agent はそこを参照/更新するようにするのが良いかと思います。
